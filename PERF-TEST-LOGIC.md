# 压测原理步骤知识逻辑文档 — WebSocket 行情 + 签名下单全链路

> 本文聚焦「为什么这样压、每一步在验证什么」的原理与逻辑，配合 README（使用说明）阅读。

---

## 一、总体思路：把交易所拆成两条被测链路

真实加密货币交易所对性能的考验来自两个截然不同的方向：

| 链路 | 协议 | 压力本质 | 关注指标 | 建模维度 |
|------|------|----------|----------|----------|
| 行情推送 | WebSocket | 海量**长连接**并发 + 高频推送 | 连接成功率、首帧延迟、消息吞吐 | **并发数（VU）** |
| 交易下单 | REST/HTTP | 高频**短请求** + 签名计算开销 | 下单延迟、成功率、验签正确性 | **到达率（RPS）** |

压测工程的第一原则：**用贴近其压力本质的执行器去建模**。行情用 `ramping-vus`（并发导向），下单用 `ramping-arrival-rate`（吞吐导向）。这也是需求 3/4/5 的设计根据。

---

## 二、第 0 步：先有靶机，再谈压测

没有稳定、可控、可观测的被测端，压测数据毫无意义。所以我们**先写 mock-server.js**：

1. **纯内置模块**：不依赖 npm，任何机器 `node scripts/mock-server.js` 即起，CI/容器零成本。
2. **手写 WebSocket**：完整走 RFC 6455 握手（SHA1+魔术串+Base64）、帧编解码（掩码/粘包/半包）、心跳，确保 k6 客户端行为被真实检验。
3. **真做验签**：签名下单端点真正重算 HMAC 并比对，能返回 Binance 风格错误码——这样「签名正确性」才有被测对象，而不是自欺欺人地永远返回 200。
4. **可注入故障**：`MOCK_ERR_RATE` 注入随机 500，用来验证客户端的重试/退避是否真的生效。

> 逻辑闭环：**客户端签名算法** ↔ **服务端验签算法** 必须严格对称，任何一端写错都会在下单成功率上暴露。

## 三、第 1 步：打通 WS 通路（需求 1）

**目的**：在加压之前，先证明「连得上、订得到、解析得对」。

- `01-ws-connect`：只验握手 `status===101`。**踩坑点**：无 pacing 时 5 个 VU 以近 6000 conn/s 疯狂建连，单线程 mock 直接被打爆，握手成功率跌到 17%。加 `sleep(0.2)` 后回到 100%——这说明**连通性冒烟必须限速**，否则测的是「建连风暴」而非「连通性」。
- `02-ws-subscribe`：`open` 发订阅 → `message` 解析。用 `parseDepth()` 做三重保护：`JSON.parse` try-catch、`Array.isArray(bids/asks)` 结构判断、`check()` 业务校验。**非深度报文（SUBSCRIBED 确认帧）返回 null 跳过**，避免误判。
- 记录 `ws_first_msg_ms`（open→首帧深度）≈ 100ms，正好等于服务端推送周期，侧面验证链路无额外积压。

## 四、第 2 步：断线重连框架（需求 2，重点）

### 4.1 为什么重连是 WS 压测的灵魂
长连接一定会断（网络抖动、服务重启、负载均衡漂移）。**一个不会重连的行情客户端在压测里毫无价值**，因为真实客户端断了会重连、会给服务端造成「重连风暴」，这恰恰是要压测的场景。

### 4.2 k6 的关键约束：ws.connect 是同步阻塞
```
调用 ws.connect(url, cb) → 阻塞 → 直到该连接关闭才返回
```
由此推出两条铁律：
- **重连循环必须在 `ws.connect` 之外**（while 循环包住它）；
- **绝不能在 socket 回调里递归 `connect`**——那会嵌套阻塞，VU 永远回不到迭代循环、并发数虚高、内存泄漏。

### 4.3 指数退避 + jitter 的数学与动机
```
delay(n) = min( base · 2^(n-1) · U[0.75,1.25],  maxDelay )
```
- `2^(n-1)`（指数）：第 1/2/3 次 = 500/1000/2000ms。间隔越拉越大，给刚恢复的服务端喘息，避免「一断就猛捶」。
- `U[0.75,1.25]`（±25% 抖动）：**核心是打散时间**。若无 jitter，500 个 VU 同一时刻断线 → 同一时刻重连 → 再次瞬时压垮 → 再同时断 → 形成**惊群/自激振荡**。加抖动后重连时刻被摊平成一段区间，负载平滑。
- `maxDelay` 封顶：防止指数无限增长导致「假死」。

实测三档退避区间 `383–617 / 790–1245 / 1536–2500 ms` 精确落在理论区间内，框架正确。

## 五、第 3 步：阶梯拉到 500 VU（需求 3）

**为什么用阶梯而不是一把拉满**：
- 阶梯（0→100→500→平台→0）能观察系统随并发上升的**性能拐点**（延迟何时开始劣化、连接何时开始失败）。
- 一把拉满只能得到一个「过/不过」的二元结论，丢失了「在多少并发下开始出问题」这一最有价值的信息。

**gracefulRampDown** 的意义：回落阶段给正在收行情的连接留出关闭时间，避免大量连接被硬切造成的误报错误率。

实测：峰值 500 VU、61 万条深度快照 100% 有效——说明 mock 与客户端在该并发下稳定。（在真实靶机上，这里就是寻找容量上限的地方。）

## 六、第 4 步：签名下单（需求 4）

### 6.1 签名算法逻辑
```
totalParams = 有序 query（业务参数 + recvWindow + timestamp）
signature   = HMAC_SHA256(secret, totalParams)  # 十六进制
请求        = POST /api/v3/order?<totalParams>&signature=<sig>   +  Header X-MBX-APIKEY
```
服务端按 `&signature=` 切割，对前半段重算 HMAC 比对——**客户端拼串顺序必须与服务端重算顺序一致**，否则必然验签失败。这是签名类压测最常见的坑。

### 6.2 为什么「实时签名」而非「纯预签名」
`pre-signer.ts` 参考了 workDemo 的 SharedArray 预加载思路，但做了升级：
- **静态部分（订单参数模板）预加载**：用 SharedArray 只解析一次、全 VU 共享，省内存、省 CPU。
- **动态部分（timestamp+signature）实时算**：Binance 的 `recvWindow` 对时间戳敏感，压测持续数分钟，纯预签名的时间戳会「过期」被 `-1021` 拒绝。实时签名保证每次请求时间戳新鲜。
- 这是「可复用的贵活预计算 + 时效敏感的活现算」的经典分层。

### 6.3 负向验证：证明验签不是摆设
只测「200 成功」不够，必须证明**错误也能被正确识别**：
- 篡改 signature → `401 / -1022`
- 错误 API Key → `401 / -2015`
- 过期 timestamp → `400 / -1021`

四类用例全部符合预期，才能说「签名链路被真正压测过」。

## 七、第 5 步：全链路混合（需求 5）

把行情（ramping-vus→500）与下单（ramping-arrival-rate→150 RPS）放进**同一个 k6 进程并发运行**，共享 `handlers.ts` 里的 exec 函数。

**为什么要混合而不是分开跑**：
- 真实系统里，行情推送和下单是**同时发生、争抢同一资源**（CPU、网络、连接池、下游服务）的。
- 分开跑各自 100% 通过，不代表混合时也行——资源竞争、GC 压力、连接饥饿只在混合负载下暴露。
- 混合场景才是「全链路」压测的真正含义。

实测：峰值 530 VU（500 行情 + ≤30 下单）、131 万 checks 100% 通过、2.1 万单 100% 成功——两条链路互不拖垮。

## 八、贯穿始终的工程原则

1. **三层防御不崩 VU**：`try-catch`（脚本级）+ `check()`（业务级）+ 空值早返回（`if (!res) return`）。任何单次异常都不该让整个 VU 挂掉。
2. **SharedArray 省内存**：k6 中每个 VU 是独立 JS VM，普通 import 会 N 份；SharedArray 共享一份，500 VU 才扛得住。
3. **指标先行**：连接率、有效率、延迟分布、重连数、验签拒绝数——先定义好 `metrics.ts`，压测结果才可解释、可门禁。
4. **环境变量解耦**：地址/密钥全走 `env.ts`，一套代码适配本地/Docker/K8s。
5. **打包 externals**：k6 内置模块由运行时注入，必须在 webpack `externals` 排除，否则 bundle 里塞进 require('k6/ws') 会运行时报错。

## 九、压测执行标准步骤（SOP）

```
1) npm install            # 装 TS/webpack 工具链
2) npm run build          # TS → dist/*.js（k6 只能跑打包产物）
3) npm run mock           # 起靶机（另一终端）
4) 冒烟：npm run test:connect / test:subscribe  # 先证通路
5) 专项：test:reconnect / test:500vu / test:order # 逐需求验证
6) 全链路：npm run test:mixed                    # 混合负载找真实表现
7) 看指标：连接率=100%? 有效率=100%? 退避区间吻合? 验签负向用例被拒?
8) 调参复跑：改 stages/rate/maxReconnects 找容量拐点
```

> 一句话总结：**先建可验签的靶机 → 打通并解析 → 会断会重连（退避+jitter）→ 阶梯加压找拐点 → 正负向验证签名 → 混合负载看真实全链路表现。**
